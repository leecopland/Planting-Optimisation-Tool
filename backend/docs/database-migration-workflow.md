# Database Migration Workflow

## Purpose

This document outlines the recommended workflow for creating and applying database schema changes using SQLAlchemy and Alembic in the Planting Optimisation Tool project.

For basic backend setup and justfile command descriptions, refer to the [backend README](../README.md).

Following this workflow helps prevent:

- unintended table drops
- schema mismatches between ORM models and the database
- missing model detection
- incorrect Alembic autogeneration output
- spatial GiST index inconsistencies
- migration conflicts and broken revisions
- inconsistent migration history, rollback issues, and non-reproducible schema changes

This documentation supplements the migration information already included in the [backend README](../README.md) by focusing specifically on safe migration practices, common pitfalls, and troubleshooting guidance.

# Recommended Migration Workflow

## 1. Make Changes to the SQLAlchemy Models

Update the required SQLAlchemy model inside:

```text
backend/src/models/
```

Examples:
- create a new table/model
- add or remove columns
- modify relationships
- add indexes
- update constraints

---

## 2. Import New Models in `models/__init__.py`

If a new model file is created, it **must** be imported in:

```python
backend/src/models/__init__.py
```

Example:

```python
from src.models.audit_log import AuditLog
from src.models.boundaries import FarmBoundary
```

### Why this matters

Alembic only detects models that are registered in SQLAlchemy metadata.

If a model is not imported into `models/__init__.py`, Alembic may fail to detect it during autogeneration. This can result in missing table creation operations for new models or incorrect schema differences for existing tables.

---

## 3. Ensure the Local Database is Up To Date

Before generating a migration, run:

```bash
just migrate
```

### Why this matters

Alembic compares the current ORM models against the current database schema.

If the local database is outdated, Alembic may compare against an incorrect baseline and generate invalid or destructive migration operations.

---

## 4. Check for Multiple Alembic Heads

Before generating or merging migrations, check whether the repository has multiple Alembic heads:

```bash
uv run alembic heads
```

### Why this matters

Multiple Alembic heads indicate that separate migration branches exist.

If multiple heads are present, developers may need to create a merge migration before generating additional revisions to avoid migration conflicts and inconsistent history.

---

## 5. Generate a Migration Revision

Generate an autogenerated Alembic migration script using:

```bash
just revision "describe changes"
```

Example:

```bash
just revision "add farm slope column"
```

This generates a new migration file inside:

```text
backend/alembic/versions/
```

---

## 6. Review the Generated Migration File

Always manually review the autogenerated migration before applying it.

Check for:
- unexpected `DROP TABLE` operations
- unintended `DROP INDEX` statements
- duplicate indexes
- incorrect column modifications
- accidental schema removals

### Example Scenarios

#### Example 1: Missing Model Import

If a new model is created but not imported into `models/__init__.py`, the autogenerated migration may be missing the expected `op.create_table(...)` operation.

Example:

```python
def upgrade():
    pass
```

This may indicate that Alembic did not detect the new model correctly.

---

#### Example 2: Unexpected Destructive Operations

Operations such as `op.drop_table(...)`, `op.drop_index(...)`, or `op.drop_column(...)` are not always incorrect, but they should match the intended schema change.

For example, a migration specifically removing a column may correctly include:

```python
op.drop_column("recommendations", "exclusions")
```

However, if Alembic generates unexpected destructive operations that were not intended as part of the schema change, the migration should be reviewed carefully before applying it.

### Important

Never assume autogenerated migrations are always correct.

Alembic only works based on differences it detects between ORM metadata and the database schema.

---

## 7. Handle Spatial GiST Indexes Correctly

If a migration creates a spatial GiST index using:

```python
op.create_index(...)
```

the same index must also exist in the model’s `__table_args__`.

Example:

```python
from sqlalchemy import Index

__table_args__ = (
    Index(
        "idx_soil_ph_geometry",
        geometry,
        postgresql_using="gist",
    ),
)
```

### Why this matters

If the index exists only in the migration but not in the ORM model, Alembic treats it as unmanaged and may attempt to remove it in future revisions.

---

## 8. Apply the Migration

After reviewing the migration:

```bash
just migrate
```

This applies the migration to the local database.

---

## 9. Test the Migration

Recommended validation and verification steps:

```bash
just test
```

This runs the existing project test suite and helps identify integration or schema-related issues, but migrations should also be reviewed and validated independently.

Additional recommended checks:
- generate ERD
- generate schema documentation
- verify tables and indexes exist correctly

---

## 10. Commit Changes Together

Always commit:
- model changes
- migration files

in the same commit or pull request.

This keeps the ORM models and migration history synchronized.

# Common Pitfalls

## Missing Model Imports

### Problem

Alembic generates unexpected `DROP TABLE` statements.

### Cause

The new model was not imported into:

```text
backend/src/models/__init__.py
```

### Solution

See [Import New Models](#2-import-new-models-in-models__init__py).

---

## Spatial GiST Index Mismatch

### Problem

Alembic attempts to remove or recreate spatial indexes repeatedly.

### Cause

The index exists in the migration but not in the ORM model definition.

### Solution

Define the same index inside the model’s `__table_args__`.

---

## Running `just revision` Before `just migrate`

### Problem

Autogenerated migration output is incorrect or inconsistent.

### Cause

The local database schema is outdated.

### Solution

Always run:

```bash
just migrate
```

before generating a new revision.

---

## Multiple Alembic Heads

### Problem

Alembic reports multiple migration heads.

Example:

```bash
uv run alembic heads
```

returns more than one head revision.

### Cause

Multiple migration branches were created independently. This commonly occurs when multiple developers generate migrations independently from the same base revision before pulling the latest migration changes from each other.

### Solution

Create a merge migration:

```bash
uv run alembic merge -m "merge heads" <head1> <head2>
```

---

## Unreviewed Autogenerated Migrations

### Problem

Unexpected destructive operations are committed accidentally.

### Cause

Migration file was not reviewed manually.

### Solution

Refer to [Review the Generated Migration File](#6-review-the-generated-migration-file) for recommended migration review steps and validation examples.

---

# Best Practices

- Always review autogenerated migrations manually
- Keep ORM models and database schema synchronized
- Use consistent naming for indexes and constraints
- Test migrations locally before pushing changes
- Commit model changes and migrations together
- Keep local databases updated before generating revisions
- Only edit autogenerated operations when needed, and ensure the final migration matches the intended schema change
- Ensure new models are imported into `models/__init__.py`

---

# Safe Migration Checklist

## Before Generating a Migration

- [ ] Model changes completed
- [ ] New models imported in `models/__init__.py`
- [ ] Local database updated using `just migrate`
- [ ] Checked for multiple Alembic heads using `uv run alembic heads`

---

## After Generating a Migration

- [ ] `upgrade()` and `downgrade()` sections reviewed manually
- [ ] No unexpected `DROP TABLE` operations
- [ ] No unintended index removals
- [ ] Spatial GiST indexes verified
- [ ] Migration naming is clear and descriptive

---

## Before Committing

- [ ] Migration applied successfully
- [ ] Tests executed successfully
- [ ] ORM models and migration files committed together

# Additional Notes

The Alembic migration template (`alembic/script.py.mako`) has been configured to automatically include:

```python
import geoalchemy2
```

This prevents missing import errors when generating migrations involving geometry columns.

## Database Initialization Scripts

The project includes database initialization scripts inside:

```text
backend/init-db/
```

These scripts run automatically during the initial container/database setup before Alembic migrations are executed.

For example, `01-remove-extensions.sql` enables required PostGIS extensions (`postgis`, `postgis_raster`) and removes unused extensions during first-time database initialization.

Developers should review these scripts when troubleshooting database setup, extension-related issues, or unexpected schema initialization behavior.
